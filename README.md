## 0. 프로젝트 개요 (간단 요약)

기존 To-Do List에서 벗어나 하루의 일정을 기사처럼 기록하고, 완료된 일정을 신문 형태로 확인할 수 있는 **신문형 일정 관리 서비스**를 기획·구현하였다. 순수 JavaScript와 LocalStorage를 활용해 기능을 구현했으며, Git 브랜치 기반의 팀 협업을 통해 설계와 개발을 진행하였다.

## 4. 핵심 설계 결정 사항 (Design Decisions)

기존 To-Do List는 완료 여부만 확인하는 경우가 많아 하루의 활동을 되돌아보기 어렵다고 판단하였다. 이에 단순한 일정 관리가 아닌 **하루를 하나의 기록으로 남길 수 있는 서비스**를 목표로 설정하고, 일정을 기사처럼 기록하여 하루를 신문 형태로 확인할 수 있도록 설계하였다. 이를 통해 사용자가 자신의 하루를 보다 직관적으로 돌아보고 성취감을 느낄 수 있는 사용자 경험을 제공하고자 하였다.

## 6. 트러블슈팅 (Troubleshooting)

### 6-1. (문제 제목)

- 증상: 일정 상태를 변경하면 칸반 보드에서는 카드가 이동했지만, 달력의 상태 점이나 상단 통계, 선택 날짜 요약에는 이전 정보가 남아 있었다.

- 원인: 칸반 보드, 달력, 통계가 각각 별도의 렌더링 함수로 구성되어 있었으며 상태 변경 시 일부 함수만 호출되고 있었다

- 해결: 일정 데이터가 변경되는 모든 기능에서 다음 순서를 공통으로 실행하도록 정리했다.

- 회고: 데이터 변경 시 모든 화면이 함께 갱신되도록 공통 렌더링 흐름을 설계하는 것이 중요하다는 점을 배웠다.

### 6-2. (문제 제목)

- 증상: 데스크톱과 모바일용 요소를 각각 만들어 숨김 처리하자 동일한 id가 중복되거나, 한쪽 요소에만 이벤트가 적용되는 문제가 발생했다. 일정 데이터가 데스크톱 화면과 모바일 화면에서 서로 다르게 보이는 현상도 나타났다

- 원인:같은 기능을 수행하는 HTML 요소를 화면 크기별로 복제했기 때문에 JavaScript에서 어떤 요소를 선택해야 하는지 명확하지 않았다. 또한 두 화면을 모두 갱신해야 하므로 렌더링 코드가 복잡해졌다.

- 해결:가능한 영역은 하나의 HTML 구조만 유지하고, 미디어 쿼리를 통해 크기와 배치를 변경했다. 부모 요소에는 Grid와 Flex 배치를 사용하고, 필요한 구간에서는 display: contents를 활용해 자식 요소를 반응형 레이아웃에 맞게 재배치했다.

- 회고: 하나의 HTML 구조를 유지하고 CSS로 반응형을 구현하는 것이 코드 관리와 유지보수에 더 유리하다는 점을 경험했다.

## 7. 자체 평가 및 회고 (Self Review)

### 7-1. 잘한 점

하나의 Todo 데이터를 달력, 칸반 보드, 통계, 신문 화면에서 공통으로 활용하는 구조를 설계했다.
신문 발간이라는 콘셉트를 통해 일정 관리 결과를 시각적으로 제공하여 서비스의 차별성을 높였다.
일정 상태 변경이 통계와 신문 화면까지 자동으로 반영되도록 데이터 흐름을 구현했다.
다크 모드와 반응형 UI를 적용하여 다양한 사용 환경을 지원했다.

### 7-2. 아쉬운 점

- 프로젝트 초기에 데이터 흐름을 충분히 설계하지 않아 렌더링 로직이 일부 중복되었다.
- 통합 과정에서 CSS 충돌과 레이아웃 수정이 발생해 예상보다 많은 시간이 소요되었다.
- 기능이 추가되면서 `app.js`가 비대해져 코드 관리와 유지보수가 어려워졌다.
- 기간 필터, 제목 정렬 등 일부 공통 기능은 개발 기간의 한계로 구현하지 못했다.
- 테스트 코드를 작성하지 못해 주요 기능을 수동으로 검증해야 했다.
- Git 브랜치와 Pull Request 사용이 익숙하지 않아 초기 협업 과정에서 충돌 해결과 코드 병합에 어려움이 있었다.

### 7-3. 다음에 개선할 점

- state, storage, calendar, board, modal 등 기능별 JavaScript 모듈 분리
- 상태 변경 이후 실행되는 공통 렌더링 함수 구성
- 기간 필터와 제목 기준 정렬 기능 추가
- Todo 등록·수정·삭제·상태 변경 테스트 작성
- 빈 문자열, 잘못된 날짜, 중복 입력 등의 예외 처리 강화
- Drag and Drop을 이용한 칸반 상태 변경 기능 도입
- CSS를 레이아웃, 컴포넌트, 테마, 반응형 파일로 분리
- 프로젝트 초기에 공통 디자인 토큰과 네이밍 규칙 정의
- 기능 통합 시점을 앞당겨 브랜치 충돌을 조기에 확인

## 9. 결론

본 프로젝트를 통해 데이터와 UI의 연동 구조, 상태 관리 흐름, 그리고 컴포넌트 간 데이터 공유 방식을 이해할 수 있었다. 또한 Git 기반 협업과 코드 통합 과정을 경험하며 협업 역량을 향상시킬 수 있었다. 최종적으로 **FlowDash**를 사용자의 하루를 신문 형태로 기록하고 관리하는 개인 일정 관리 서비스로 완성하였다.
